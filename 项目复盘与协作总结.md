## 复盘一：迭代开发经验（主要给我——AI）

### ✅ 做对的

1. **单一真相源（Single Source of Truth）建得早**：`data/controls.js` / `hardware.js` / `cases.js` / `quiz.js` 集中管数据，页面用模板按 id 渲染。这让"加一个控件/能力"变成填数据，而不是复制页面。
2. **范式复用**：控件详情页和硬件详情页共用同一套"4-Tab + 演示组件"，学习和维护成本都低。
3. **提交规范执行到位**：四段式 commit（原因/方案/测试/问题）后期贯彻得好，历史可读性强。

### ⚠️ 踩的坑 → 根因 → 固化动作

| 坑 | 根因 | 以后怎么做（原则） |
|---|---|---|
| WXSS 中文类名编译失败（50f0d0d） | 用中文数据字段动态拼 class（`sim-{{中文}}`） | **铁律：选择器名只能 ASCII。数据用英文 token，显示文字与 class 分离。** |
| 详情页空白（多次） | `lazyCodeLoading` 与新增页面组件注入冲突 | 引入平台级配置项前，先查它对"动态/新增组件"的副作用；改完立刻真机+模拟器双验证。 |
| 死代码残留（案例精简后 case-demos 没清） | 删数据时没顺藤摸瓜删渲染分支 | **删除任何实体，必须全链路搜引用**：数据 → 渲染分支 → 样式 → 跳转 → 题库关联。 |
| 跳转路由错（硬件题跳到控件页） | 路由写死单一路径，没考虑 id 归属 | 跨类型跳转必须按 id 归属分派；改完扫一遍所有 relatedId 是否命中有效目标。 |
| AppSecret 进了 git 历史 | 真实密钥写进了会提交的文档 | **任何会入仓的文件，写之前先假设它会公开**。密钥/路径/AppID 一律用占位符。 |

### 核心心法（给我自己的）
- **"填数据"式扩展 > "复制页面"式扩展**——先建数据层和模板，再堆内容。
- **删除比新增更危险**——新增只影响新代码，删除会在四处留下悬空引用。
- **改完就自检**：每次结构性改动后，跑一遍"引用一致性"检查再交付，不要等用户发现空白页。

## 复盘二：协作沟通效率（给你，也给我）

你这次提的最关键的一点是：**"你可以多思考多提问，这样就能更快速完成开发。"** 完全成立。复盘我们的轮次，很多返工其实是"沟通前置不足"造成的，本可以用 1~2 个问题避免。

### 三个高频返工，都能靠"先问一句"消除

1. **IA / 需求边界**：首页做了"案例/硬件"入口 → 你说多余要删；案例做了 11 个 → 你说精简到 8。
   → **教训：涉及信息架构、数量、范围的决策，动手前我应先给 2~3 个方案让你拍板，而不是先做一版再改。**

2. **运行环境假设**：我默认在 C 盘副本改，真仓其实在 D 盘 → 一次返工 + 清理。
   → **教训：每个新会话/新任务开始前，我先确认"在哪个目录、哪个分支、有没有 remote"，得到确认再动手。**

3. **排查方向**：模拟器启动失败，我先查了 28 个 JSON/appid/语法（半天），结果是开发者工具缓存。
   → **教训：现象类问题先做"最便宜的排除"（清缓存、重启、换网络），再深挖代码。**

### 我们已经磨合出的默契（继续保持）
- 你偏好：**结论先行、分点、极简方案、做完再迭代精简**（如主动去掉赞助区、双 docs）。
- 你偏好：**四段式 commit + 完整 body**、**详细调用链 + 文件行号引用**。
- 你偏好：**让我自己推导，不轻易反问**——所以我要提的问题必须是"真正影响方向、我推不出来"的，而不是偷懒式确认。

### 给我的提问纪律（关键平衡）
> 你不喜欢被反复打断确认，但欢迎"影响方向的关键提问"。所以规则是：
> - **能自己查/推导的 → 直接做**（目录结构、API 用法、实现细节）。
> - **影响架构/范围/不可逆的 → 先问**（IA、删改范围、平台选型、数据契约）。
> - 提问时**带上我的建议选项**，让你只需选择，而不是从零描述。

## 复盘三：开发流程沉淀（用流程/原则驱动，减少选择）

把上面的教训固化成 4 张清单。下个项目直接照着走，不再临场决策。

### 清单 A · 项目启动前（对齐，避免大返工）
- [ ] 确认工作目录 + 分支 + 是否有 remote
- [ ] 确认交付物形态（应用/文档/网页）与最终托管/部署方式
- [ ] 确认信息架构（页面/Tab/模块划分）——用图或列表让用户拍板
- [ ] 确认范围与优先级（哪些必做、哪些占位、数量上限）
- [ ] 确认脱敏要求（密钥/AppID/路径/真实信息如何处理）
- [ ] 建立单一真相源（数据层）+ 页面模板，再开始堆内容

### 清单 B · 每次结构性改动后（一致性自检）
- [ ] 删除的实体：数据 / 渲染分支 / 样式 / 跳转 / 关联题目 全部清理
- [ ] 新增/改名：全局搜旧名，确认无残留
- [ ] 跳转路由：所有 relatedId / 目标页命中有效目标
- [ ] 语法校验：`node --check` 全 .js + 全 .json 合法
- [ ] 模拟器 + （必要时）真机 双验证

### 清单 C · 提交规范
- [ ] 四段式：【原因】【方案】【测试】【问题单号】，完整多行 body
- [ ] 相关改动一次性提交，不留半成品
- [ ] 提交前确认无密钥/隐私入仓

### 清单 D · 静态站/部署（docsify 专项）
- [ ] 站点在子目录时，内链用裸相对路径，**不带 `docs/` 前缀**
- [ ] 有下划线开头文件（`_sidebar`/`_coverpage`）→ **必放 `.nojekyll`**
- [ ] docsify 必须走 http:// 访问，不能双击 `file://` 打开
- [ ] GitHub Pages Source 选对目录，推送后等 1~2 分钟并强刷验证

### 五条通用原则（凌驾于清单之上）
1. **想清楚再动手**：影响方向的先对齐，实现细节的直接做。
2. **单一真相源**：数据集中，页面靠模板+id 渲染。
3. **删除即排雷**：删任何东西先全链路搜引用。
4. **改完即自检**：结构性改动后跑一致性检查再交付。
5. **入仓即公开**：任何会提交的文件都当作会被公开来写。

## 附：下一个项目的第一步

新项目启动时，我会**主动**做这两件事，不用你提醒：
1. 先跑一遍【清单 A】，把没确认的点用"带建议选项的提问"一次性抛给你；
2. 建好数据层 + 模板骨架后，再和你确认一次方向，然后进入快速堆量阶段。

> 这样我们的协作节奏会变成：**一次对齐 → 快速产出 → 一次验收**，而不是 **边做边改 → 反复返工**。
